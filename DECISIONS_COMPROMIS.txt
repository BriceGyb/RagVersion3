--- PHASE 1 : REPRODUCTION DE L'ARTICLE (TERMINE) ---

ETAPE A — Construction du pipeline RAG
  - Chargement de la base MITRE ATT&CK (enterprise-attack.json, 697 techniques)
  - Chunking hierarchique : chunks parents 2000 chars, enfants 500 chars
  - Embeddings : nomic-ai/nomic-embed-text-v1.5 (HuggingFace)
  - Base vectorielle : FAISS (en memoire, non sauvegardee sur disque)
  - Retriever hybride : FAISS (dense) + BM25 (lexical) + FlashrankRerank
  - 4 LLMs charges : GPT-4o mini, GPT-3.5 turbo, BART base, Flan-T5 small

ETAPE B — Generation des reponses LLM (TERMINE)
  - 229 requetes CTI-ATTACK traitees par 4 LLMs
  - 916 reponses generees, sauvees dans RAG-evaluation-results.xlsx
  - Duree : ~2h25

ETAPE C — Evaluation RAGAS du RAG (TERMINE)
  - Nombreux crashs : quota API, timeout, bugs RAGAS 0.4.x
  - Solution finale : script avec checkpoint toutes les 20 lignes
  - Compromis appliques (voir section DECISIONS ci-dessous)
  - Fichiers : RAG-evaluation-results-final.xlsx, ragas_scores_by_llm.xlsx

  SCORES FINAUX RAG (GPT-4o, CTI-ATTACK, 229 requetes) :
    context_recall    = 0.4192  (article : 0.418) -> quasi identique
    context_precision = 0.2416  (article : 0.340) -> ecart ~0.10
    answer_relevancy  = 0.2455  (article : 0.684) -> ecart important
    noise_sensitivity = 0.0126  (article : 0.023) -> proche

--- PHASE 2 : IMPLEMENTATION MCP (TERMINE — EN COURS D'EVALUATION) ---

ETAPE D — Serveur MCP MITRE ATT&CK (TERMINE)
  Fichier : mcp_mitre_server.py
  15 outils exposes en 3 couches :

  Couche 1 — Comprehension de la requete :
    - decompose_query     : decompose en sous-questions
    - extract_keywords    : extrait termes techniques
    - identify_context_type : identifie la tactique probable

  Couche 2 — Recuperation MITRE :
    - search_techniques      : recherche semantique
    - get_technique          : detail par ID
    - get_techniques_by_tactic : techniques par tactique
    - get_subtechniques      : sous-techniques
    - get_group              : groupes de menace
    - get_software           : logiciels/malwares
    - get_mitigations        : mitigations

  Couche 3 — Raffinement et Agregation :
    - cross_reference        : relations entre techniques
    - filter_by_platform     : filtre par plateforme
    - rank_candidates        : classe par pertinence
    - validate_technique     : valide et score
    - format_final_answer    : formate la reponse finale

ETAPE E — Inference MCP (TERMINE)
  Fichier : run_mcp_agent.py
  - Modele : gpt-4o-mini (meme que RAG pour comparaison equitable)
  - 229 requetes CTI-ATTACK traitees
  - 228/229 reponses valides (1 erreur)
  - Moyenne 11.5 appels d'outils par requete
  - Checkpoint toutes les requetes
  - Resultats : mcp_agent_results.xlsx, mcp_checkpoint.csv

--- PHASE 3 : EVALUATION RAGAS MCP (TERMINE) ---

ETAPE F — Evaluation RAGAS sur resultats MCP
  Fichier : run_ragas_mcp.py
  - Memes 4 metriques que le RAG :
      context_recall, context_precision, answer_relevancy, noise_sensitivity
  - Juge LLM : gpt-4o-mini (identique au RAG)
  - 229 echantillons (1 seul modele : mcp-gpt-4o-mini)
  - Checkpoint actif (ragas_mcp_checkpoint.csv, ragas_mcp_checkpoint_context.csv)
  - Sortie : mcp_evaluation_results.xlsx, mcp_ragas_scores.xlsx

  SCORES FINAUX MCP (gpt-4o-mini, CTI-ATTACK, 229 requetes) :
    context_recall    = 0.4156
    context_precision = 0.3314
    answer_relevancy   = 0.5783
    noise_sensitivity  = 0.0732

--- PHASE 4 : COMPARAISON ET RAPPORT ---

ETAPE G — Tableau comparatif RAG vs MCP (TERMINE)

  +--------------------+-----------+-----------+-------------+
  | Metrique           | RAG       | MCP       | Difference  |
  +--------------------+-----------+-----------+-------------+
  | context_recall     | 0.4192    | 0.4156    | -0.0036     |
  | context_precision  | 0.2416    | 0.3314    | +0.0898     |
  | answer_relevancy   | 0.2455    | 0.5783    | +0.3328     |
  | noise_sensitivity  | 0.0126    | 0.0732    | +0.0606     |
  +--------------------+-----------+-----------+-------------+

  Analyse rapide :
  - context_recall : quasi identique, les deux approches retrouvent
    une proportion similaire des techniques pertinentes.
  - context_precision : le MCP est plus precis (+9pts) car les outils
    de recherche ciblee (search_techniques, get_techniques_by_tactic)
    filtrent mieux que le retriever FAISS+BM25 generique.
  - answer_relevancy : ecart majeur (+33pts, MCP x2.3 superieur).
    L'agent MCP raisonne par etapes (decompose -> search -> validate
    -> format_final_answer) ce qui produit des reponses mieux
    structurees et alignees sur la question posee.
  - noise_sensitivity : le MCP est plus sensible au bruit (+6pts,
    moins bon). Hypothese : les 11.5 appels d'outils en moyenne
    introduisent davantage de contexte intermediaire (resultats
    d'outils non filtres) qui peut distraire le modele lors de la
    generation finale.

==============================================================
DECISIONS ET COMPROMIS
==============================================================

DECISION 1 : UN SEUL LLM EVALUE (GPT-4o mini)
  - BART et Flan-T5 exclus : resultats nuls ou trop faibles
  - GPT-3.5 exclu : garder seulement le meilleur modele
  - Impact : 229 echantillons au lieu de 916
  - Justification : comparaison RAG vs MCP plus propre avec meme modele

DECISION 2 : 4 METRIQUES RAGAS (sur 6 dans l'article)
  Retenues  : context_recall, context_precision, answer_relevancy, noise_sensitivity
  Supprimees : faithfulness, factual_correctness (trop lentes, scores bas)
  Impact    : ~1400 appels API au lieu de ~5500

DECISION 3 : CONTEXT_RECALL ET PRECISION CALCULES 1x (GLOBAL)
  Ces metriques evaluent le retriever, pas le LLM.
  Calculees une seule fois sur les 229 requetes. (Idem article page 15)

DECISION 4 : VERSION RAGAS 0.4.3 (article utilise 0.2.13)
  Difference de nom de colonne : "noise_sensitivity(mode=relevant)"
  Impact : scores potentiellement legerement differents de l'article

DECISION 5 : JUGE RAGAS = gpt-4o-mini
  Meme modele pour les deux evaluations (RAG et MCP) pour coherence.

DECISION 6 : CHECKPOINT AUTOMATIQUE
  Sauvegarde toutes les 20 lignes (RAGAS) ou toutes les requetes (MCP).
  Permet de reprendre sans repartir de zero en cas de crash.

DECISION 7 : MEME MODELE POUR RAG ET MCP (gpt-4o-mini)
  Garantit que la difference de performance est due a la methode
  (RAG passif vs MCP actif) et non au modele.
